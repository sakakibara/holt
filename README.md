# holt

`holt` is a workspace CLI that gives each project one home for its code (git
repos), its cloud-synced docs/assets/links, and a single view over both.

- **One shared clone per repo.** A repo used by many projects is cloned once at
  a canonical `~/Code/<host>/<owner>/<repo>` path and referenced everywhere,
  never duplicated per project.
- **Derived, regenerable view.** The browsable hub holds no state of its own;
  `holt sync` rebuilds it idempotently from per-project markers, so it is
  always safe to discard and regenerate.
- **Backend-agnostic.** Any cloud or sync tool works via a named preset or a
  direct path - nothing is presumed and nothing is auto-detected.
- **Shell-native.** `h`/`hi` navigation plus dynamic tab-completion for fish,
  zsh, bash, and PowerShell.
- **Safe by construction.** Atomic marker/config writes, per-project locking,
  and clones that are never deleted except by an explicit, safety-gated prune.

## How it works: three trees

Every project spans three trees, each kept in the shape its job needs:

```
CODE      ~/Code/<host>/<owner>/<repo>          local, git-safe, one shared clone per repo
CONTENT   <synced>/projects/<org>/<project>/    cloud-synced, pure files, the source of truth
HUB       ~/Projects/<org>/<project>/           local, derived view, regenerable
```

**CODE** - one shared clone per repo, at `~/Code/<host>/<owner>/<repo>`. Cloned
once and referenced by every project that uses it, never project-local.
Identity keys on the literal host in the URL, so reach a shared repo by its
canonical host, not an `~/.ssh/config` alias (which would key a second clone).

**CONTENT** - the cloud-synced source of truth. Holds `docs/`, `assets/`,
`links/`, and a `.holt.json` marker (the project's org, name, and member
repos). Pure files, never symlinks - some backends (Google Drive) reject them.

**HUB** - where you browse a project whole. Every entry is a symlink:

```
~/Projects/<org>/<project>/
  docs   -> <synced>/projects/<org>/<project>/docs
  assets -> <synced>/projects/<org>/<project>/assets
  links  -> <synced>/projects/<org>/<project>/links
  code/<repo> -> ~/Code/<host>/<owner>/<repo>     (one per member repo)
```

The hub holds no state of its own - it is derived from CONTENT's markers,
regenerated by `holt sync` or rebuilt from scratch by `holt restore`. Its
symlinks are cattle, not pets: never hand-edited, always regenerated.

A repo's canonical path is always the real `~/Code/...` one. Hub `code/<repo>`
symlinks are for browsing only - `holt path`, the shell shortcuts, and anything
built on holt resolve to the real path, so a shared repo has one unambiguous
path however you reach it.

## Worktrees

To have two branches of a repo checked out at once (say, to compare them),
`holt worktree` wraps `git worktree` - an extra checkout that shares the repo's
one clone, never a second clone:

```
holt worktree acme/backend feature/new-auth   # create; prints its path
holt worktree acme/backend                     # list
holt worktree acme/backend feature/new-auth -r # remove
```

Worktrees live in a sibling `<clone>@worktrees/` dir and surface in the hub as
a single `code/<repo>@worktrees` link, so a project opened at its hub root
(an editor, an agent) reaches every branch's checkout at
`code/<repo>@worktrees/<branch>`. git owns the tree inside that link, so slashy
branch names (`feature/new-auth`) just nest, with no flattening. Navigate to
one with `h <project>/<repo>@<branch>` (the branch tab-completes). `archive
--prune` never reclaims a clone that has worktrees, since they may hold
uncommitted work.

## Install

```sh
curl -fsSL https://raw.githubusercontent.com/sakakibara/holt/main/scripts/install.sh | sh
```

Installs the latest release to `~/.local/bin/holt`, verifying the download
against the release checksums. Pin a version or change the location with the
`HOLT_VERSION` / `HOLT_INSTALL_DIR` environment variables, or `--version` /
`--dir` flags after `sh -s --`:

```sh
curl -fsSL https://raw.githubusercontent.com/sakakibara/holt/main/scripts/install.sh \
  | sh -s -- --version v0.1.0 --dir /usr/local/bin
```

Prefer a manual install? Grab the tarball for your platform from the
[latest release](https://github.com/sakakibara/holt/releases/latest)
(`holt-{macos,linux}-{aarch64,x86_64}.tar.gz`) and put `holt` on your `PATH`, or
build from source with Zig 0.16 (`zig build`, binary at `zig-out/bin/holt`).
Either way, `holt upgrade` self-updates afterward.

## Quick start

```sh
holt setup                                 # choose where your cloud syncs
holt new personal/notes                    # create a project
holt add notes github.com/you/notes.git    # attach a repo (clones it)
holt status notes                          # see its git state
```

Add the [shell integration](#shell-integration) for `h`/`hi` navigation and tab
completion.

## Config

holt reads `~/.config/holt/config.toml` (or `$XDG_CONFIG_HOME/holt/config.toml`).
There is no auto-detection: without a config file, any workspace command stops
and points you at `holt setup`.

Only one value varies per user - `synced_root`, the folder your cloud syncs.
`code_root` (`~/Code`) and `hub_root` (`~/Projects`) are machine-local. Set
`synced_root` directly, or through a named `backend` preset:

```toml
# Direct
[workspace]
synced_root = "~/some/folder"
code_root   = "~/Code"
hub_root    = "~/Projects"
```

```toml
# Or a named preset
[workspace]
backend   = "dropbox"
code_root = "~/Code"
hub_root  = "~/Projects"

[backends.dropbox]
synced_root = "~/Dropbox/workspace"
```

A backend name is free-form: any string works as long as a matching
`[backends.<name>]` exists. Nothing is baked in - any cloud works, whether or
not holt ships a seed for it. Each `synced_root` is a holt-owned `workspace`
subfolder inside the cloud (holding `projects/` beside any siblings you keep,
like `archive/` or `backups/`), not the cloud's root.

### Seeded presets

`holt setup` seeds a starter set into your file - never consulted at runtime,
just a head start you can edit or delete. Clouds with a predictable path are
ready to use; account-specific ones are seeded commented-out for you to fill in
once:

| Ready to use | Fill in first |
| --- | --- |
| `icloud` `dropbox` `nextcloud` `mega` `pcloud` | `gdrive` `onedrive` `proton` `box` |

### Config commands

| Command | Does |
| --- | --- |
| `holt setup` | Create the config - interactive, or scriptable with `--backend <name>` / `--synced-root <path>` (plus `--code-root`, `--hub-root`, `--force`) |
| `holt backends` | List every preset with its resolved path, whether it exists, and which is active |
| `holt backend [<name>]` | Show the active backend, or switch to `<name>` (surgical edit; comments preserved) |
| `holt config [edit]` | Show the resolved config, or open it in `$EDITOR` |

Presets are plain TOML you edit by hand or via `holt config edit`; holt reads
`[backends.*]` blocks but never rewrites them.

## Shell integration

Source holt's generated snippet for tab-completion (subcommands, flags, and live
values - projects, orgs, member repos, archived projects, backends) plus the `h`
(jump to a project or repo) and `hi` (fuzzy-pick a project) functions:

```fish
holt init fish | source
```
```zsh
eval "$(holt init zsh)"
```
```bash
eval "$(holt init bash)"
```
```powershell
holt init powershell | Invoke-Expression
```

## Selecting a project

Most commands take a `<project>` selector, matched against each project's
`<org>/<name>` and bare `<name>` in descending tiers - the first tier with any
match decides:

| Tier | Example |
| --- | --- |
| **Exact** | `personal/dotfiles` or `dotfiles` |
| **Prefix** | `dot` matches `dotfiles` |
| **Substring** | `file` matches `dotfiles` |
| **Fuzzy** | `dtf` matches `dotfiles` by subsequence; tightest span wins |

Matching is smartcase (case-insensitive unless the selector has an uppercase
letter). An ambiguous selector resolves to nothing and prints the candidates so
you can pick one. Two exceptions:

- `new` and `rename` need the full `<org>/<name>` - they create a project, so
  there is nothing to abbreviate yet.
- A `<repo>` argument (in `rm`, `alias`, `run --repo`, and `<project>/<repo>`
  paths) is a member-repo short name within the resolved project, not a project
  selector.

## Commands

`<project>` is a selector (see [Selecting a project](#selecting-a-project));
`<org>/<name>` means the full form is required. Run `holt help` for the full
list, or `holt <command> --help` for one command's usage.

**Set up**

| Command | Description |
| --- | --- |
| `holt version` | Print the holt version |
| `holt init <fish\|zsh\|bash\|powershell>` | Print shell integration code (`h`/`hi` functions) |
| `holt setup [--backend <name>\|--synced-root <path>] [--force]` | Create `config.toml`, picking or seeding a backend |
| `holt backends` | List configured backend presets |
| `holt backend [<name>]` | Show or switch the active backend preset |
| `holt config [edit]` | Show the resolved configuration, or edit it in `$EDITOR` |

**Projects**

| Command | Description |
| --- | --- |
| `holt new <org>/<name> [url]` | Create a project, optionally cloning its first repo (`url` accepts `owner/repo` shorthand) |
| `holt rename <old> <new-org>/<new-name>` | Rename a project, moving its content and rebuilding its hub |
| `holt org rename <old-org> <new-org> [--yes]` | Rename an org, moving every project in it |
| `holt archive <project> [--prune] [--yes]` | Move a project's content to `archive/` and drop its hub; `--prune` also reclaims member clones that are clean, synced, and unused by any active project |
| `holt restore --all [-j N] \| <project>` | Clone every missing repo (in parallel, deduped by clone) and rebuild all hubs, or unarchive one project |
| `holt delete <project> [--yes]` | Delete a project's content and hub (clones kept); type `org/name` to confirm unless `--yes` |
| `holt backup <project>` | Tar a project's content dir into synced `backups/` |
| `holt info <project> [--json]` | Show a project's paths and member repos |

**Repos**

| Command | Description |
| --- | --- |
| `holt add <project> <url>` | Add a repo to a project, cloning it if absent (`url` accepts `owner/repo` or `host/owner/repo` shorthand) |
| `holt get <url> [--update]` | Clone a repo standalone into the code tree (no project attached; accepts shorthand) |
| `holt rm <project> <repo>` | Remove a repo from a project (the shared clone stays on disk) |
| `holt adopt <project> <path> [--force]` | Register an existing clone into a project, moving it to its identity path |
| `holt promote <repo> [--dry-run] [--yes] [--force]` | Move a local repo to its real remote identity once it has an origin (previews and confirms unless `--yes`/`--force`) |

**Navigate & inspect**

| Command | Description |
| --- | --- |
| `holt path [<project>\|<project>/<repo>]` | Print the filesystem path for a project or project/repo |
| `holt list [--paths] [--org <org>] [--json]` | List every project in the workspace |
| `holt status [<project>] [--dirty] [--org <org>] [--json]` | Show git status across a project's (or every project's) member repos |
| `holt recent [-n N] [--org <org>] [--json]` | List projects ordered by their most recent commit |
| `holt edit <project>` or `holt edit <project>/<repo>` | Open a project's docs in `$EDITOR`, or a member repo's clone |
| `holt run (<project>\|--org <org>\|--all) [--repo <name>] -- <cmd...>` | Run a command in each member repo's clone (deduped by real clone path) |
| `holt worktree <project>/<repo> [<branch>] [--remove]` | Create, list, or remove a repo's git worktrees (extra branch checkouts of its one clone) |

**Maintain**

| Command | Description |
| --- | --- |
| `holt sync [--dry-run]` | Reconcile every project's hub with its marker |
| `holt doctor [--fix] [--full] [-j N]` | Check the workspace for invariant violations and drift |
| `holt upgrade [<version>] [--yes]` | Install the latest holt release, or a specific `<version>` |

## Scripting

`holt list --json` and `holt status --json` emit a JSON array to stdout - no
color, never the human empty-state prose (an empty workspace is `[]`). Both
compose with `--org`.

```
$ holt list --json
[{"org":"acme","name":"widget","hub_path":"/Users/me/Projects/acme/widget","content_path":"/Users/me/Dropbox/workspace/projects/acme/widget","repos":["widget-repo"]}]

$ holt status --json
[{"project":"acme/widget","repos":[{"name":"widget-repo","state":"clean","branch":"main"}]}]
```

`state` is one of `clean`, `dirty`, `unpushed`, `no-upstream`, `missing`, or
`unreadable`.

Without `--json`, a few outputs are still a stable contract:

- `holt path <project>` prints one absolute hub path; `holt path
  <project>/<repo>` prints the repo's real clone path; `holt path` alone prints
  the hub root.
- `holt list` prints one `org/name` per line; `--paths` appends a tab and the
  hub path.
